# Unsupported Features

**Last Updated:** 2026-06-28
**Version:** 1.0

This document explicitly lists features and capabilities that are NOT provided by the SOC Analyst Agent. This prevents misunderstandings about the agent's scope and avoids deployment failures caused by expecting functionality that does not exist.

---

## 1. Real-Time Packet Capture Analysis

**Status:** Not Supported
**Reason:** The SOC Analyst Agent operates at the alert and log analysis layer, not at the network packet layer. Packet capture (PCAP) analysis requires specialized tooling (Zeek, Suricata, Wireshark, Moloch/Arkime) with direct access to network taps or SPAN ports.

**What the agent does instead:** The agent consumes network-related alerts generated by SIEM detection rules that may originate from network monitoring tools. It correlates these alerts with threat intelligence but does not inspect raw packets.

**Workaround:** Deploy a dedicated network security monitoring (NSM) solution and configure it to forward alerts to the connected SIEM. The agent will process those alerts through the normal pipeline.

**Future Roadmap:** No plans to add packet capture analysis. This remains outside the agent's architectural scope.

---

## 2. Endpoint Remediation

**Status:** Not Supported (Containment Only)
**Reason:** The agent supports endpoint containment actions (isolating a host via CrowdStrike Falcon's network containment API) but does not perform endpoint cleanup, malware removal, registry repair, or system restoration.

**What the agent does:**
- Trigger host network containment via CrowdStrike API (requires SOC lead approval)
- Recommend remediation actions in incident reports
- Track containment status and lift containment when approved

**What the agent does NOT do:**
- Delete or quarantine malicious files on endpoints
- Kill malicious processes
- Restore modified registry entries
- Roll back system changes
- Deploy patches or updates
- Run forensic collection tools on endpoints

**Workaround:** Use CrowdStrike Falcon's Real Time Response (RTR) or your EDR platform's remediation features directly. The agent's incident report includes recommended remediation steps that a human analyst or automated playbook can execute.

**Future Roadmap:** Guided remediation workflows (with human approval gates) are under consideration for a future release.

---

## 3. Custom YARA Rule Execution

**Status:** Not Supported
**Reason:** YARA rule scanning requires access to file system contents or memory dumps, which the agent does not have. YARA execution is a resource-intensive operation best handled by endpoint agents or dedicated malware analysis sandboxes.

**What the agent does instead:**
- Matches IOCs (hashes, IPs, domains) against VirusTotal, AbuseIPDB, and MISP
- Maps alerts to MITRE ATT&CK techniques
- Consumes YARA-based detection results from SIEM rules that already use YARA (e.g., Elastic's YARA integration)

**Workaround:** Configure YARA scanning in your SIEM or EDR platform. Results from YARA-based detection rules will flow into the agent's alert pipeline automatically.

**Future Roadmap:** Planned for v2.x. The agent will support submitting file hashes to a YARA scanning service and correlating results, but will not execute YARA rules locally.

---

## 4. Automated Incident Containment Without Approval

**Status:** Not Supported (By Design)
**Reason:** Automated containment actions (host isolation, account lockout, firewall rule changes) without human approval create unacceptable risk of business disruption. A false positive leading to automated containment of a production server or executive account could cause significant operational impact.

**What the agent does instead:**
- Generates containment recommendations with severity and confidence scores
- Sends approval requests to designated SOC leads via PagerDuty, Slack, or email
- Executes containment only after explicit approval from an authorized user
- Logs all containment actions with the approving user, timestamp, and justification

**Workaround:** None. This is a deliberate design decision. Organizations requiring fully automated containment should implement SOAR playbooks separately and use the agent's alert data as input.

**Future Roadmap:** Auto-containment with configurable confidence thresholds may be added behind a feature flag in a future release, restricted to the `admin` role and requiring explicit opt-in per action type.

---

## 5. Log Source Onboarding

**Status:** Not Supported
**Reason:** Configuring log collection agents, forwarders, syslog pipelines, or cloud log connectors is outside the agent's scope. This is a SIEM administration task.

**What the agent does instead:** Consumes alerts generated by detection rules in the connected SIEM, regardless of the underlying log sources that feed those rules.

**Workaround:** Use SIEM vendor documentation and tools (Splunk Universal Forwarder, Elastic Agent, Azure Monitor Agent) to onboard log sources.

---

## 6. SIEM Detection Rule Authoring

**Status:** Not Supported
**Reason:** Writing and tuning SIEM detection rules requires deep knowledge of the organization's environment, data schema, and threat model. This is a specialized skill that the agent does not automate.

**What the agent does instead:**
- Reports false positive rates per detection rule to help analysts identify rules needing tuning
- Maps existing alerts to MITRE ATT&CK to identify coverage gaps
- Provides alert trend analysis showing which rules generate the most noise

**Workaround:** Use the agent's analytics data to prioritize rule tuning efforts in your SIEM's rule editor.

---

## 7. Digital Forensics and Evidence Collection

**Status:** Not Supported
**Reason:** Forensic evidence collection (disk imaging, memory acquisition, chain-of-custody management) requires physical or agent-level access to affected systems and strict procedural controls that are beyond the scope of an automated analysis agent.

**What the agent does instead:** Generates investigation summaries with timeline reconstruction, IOC lists, and affected asset inventories that support forensic investigations.

**Workaround:** Use dedicated DFIR tools (Velociraptor, GRR, The Sleuth Kit) triggered by the agent's incident reports.

---

## 8. User and Entity Behavior Analytics (UEBA)

**Status:** Not Supported
**Reason:** UEBA requires long-term behavioral baseline modeling with access to identity, authentication, and activity logs over extended periods. This is typically a feature of the SIEM platform itself (Splunk UBA, Elastic ML, Sentinel UEBA).

**What the agent does instead:** Processes UEBA-generated alerts from the connected SIEM. The agent enriches these alerts and correlates them with other indicators but does not perform its own behavioral modeling.

---

## 9. Compliance Reporting

**Status:** Not Supported
**Reason:** Generating compliance reports (SOC 2, PCI DSS, HIPAA, GDPR, ISO 27001) requires mapping security controls to regulatory frameworks, which is a governance function outside the agent's scope.

**What the agent does instead:** Provides operational metrics (mean time to detect, mean time to respond, alert volumes, false positive rates) that can be used as inputs to compliance reports.

---

## 10. Multi-Tenant Deployment

**Status:** Not Supported in Current Version
**Reason:** The agent currently operates as a single-tenant deployment, connecting to one set of SIEM instances and TI sources. Multi-tenant isolation (separate data, configurations, and access controls per tenant) is not implemented.

**Workaround:** Deploy separate agent instances per tenant, each with its own database, Redis instance, and configuration.

**Future Roadmap:** Multi-tenant support with tenant-level data isolation is planned for v3.0.
