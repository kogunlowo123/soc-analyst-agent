# Assumptions

**Last Updated:** 2026-06-28
**Version:** 1.0

This document lists every assumption the SOC Analyst Agent makes about the deployment environment, infrastructure, and external services. If any assumption does not hold, the agent will not function correctly until the gap is resolved.

---

## 1. SIEM Infrastructure

### 1.1 SIEM Is Deployed and Accessible via API

The SOC Analyst Agent does not deploy, install, or configure any SIEM platform. At least one of the following SIEM platforms must be operational and exposing its API before the agent is deployed:

- **Splunk Enterprise / Splunk Cloud** with REST API enabled on port 8089 (management port) or via Splunk Cloud API endpoint.
- **Elastic Security** (Elasticsearch + Kibana) with the detection engine enabled and API accessible on the configured Kibana port (default 5601).
- **Microsoft Sentinel** with an active Log Analytics workspace and the `Microsoft.SecurityInsights` resource provider registered on the Azure subscription.

**What happens if violated:** The agent's SIEM connector service will fail health checks and enter a crash loop. Alerts will not be ingested. The agent dashboard will show "SIEM Disconnected" status.

### 1.2 SIEM Service Account Is Provisioned

A dedicated service account (or API key) with appropriate read/write permissions must be created in each connected SIEM before agent deployment.

| SIEM | Required Permissions |
|------|---------------------|
| Splunk | `search` capability, `list_search_head` for distributed environments, access to relevant indexes |
| Elastic | `superuser` or custom role with `read` on `.siem-signals-*` indices, `all` on detection rules, `read` on relevant data indices |
| Sentinel | `Microsoft Sentinel Responder` Azure role on the resource group containing the workspace |

**What happens if violated:** API calls will return 401 or 403 errors. The agent will log authentication failures and mark the SIEM integration as degraded.

### 1.3 SIEM Data Sources Are Configured

The SIEM must already be receiving log data from organizational sources (firewalls, endpoints, identity providers, cloud trails, etc.). The agent analyzes alerts generated by SIEM detection rules; it does not configure log ingestion pipelines.

**What happens if violated:** The agent will connect successfully but will have no alerts to process. Detection coverage will appear as zero.

## 2. Threat Intelligence Services

### 2.1 API Keys Are Provisioned

Valid API keys must be obtained and provisioned for each threat intelligence service before deployment:

| Service | Key Provisioning | Minimum Tier |
|---------|-----------------|-------------|
| VirusTotal | API key from virustotal.com account | Free (500 req/day) for evaluation; Premium (30K req/day) for production |
| AbuseIPDB | API key from abuseipdb.com account | Free (1,000 checks/day) for evaluation; Premium (50K checks/day) for production |
| MISP | API key from MISP instance user settings | N/A (self-hosted) |
| CrowdStrike | OAuth2 client credentials from Falcon console | Falcon Go or higher |

**What happens if violated:** IOC enrichment will fail for the affected service. The agent will log warnings and continue processing with available sources. Enrichment data will be incomplete.

### 2.2 MISP Instance Is Self-Managed

If MISP integration is enabled, the organization operates its own MISP instance (or subscribes to a MISP community). The agent connects to MISP as a consumer of threat intelligence data; it does not deploy or manage MISP.

**What happens if violated:** MISP-related enrichment features will be unavailable. The agent will skip MISP queries and rely on other TI sources.

## 3. Network Connectivity

### 3.1 Agent-to-SIEM Network Path Exists

The agent's Kubernetes pods must have network connectivity to all configured SIEM API endpoints:

- Splunk: TCP port 8089 (REST API) or HTTPS 443 (Splunk Cloud)
- Elastic: TCP port 5601 (Kibana API) and 9200 (Elasticsearch API)
- Sentinel: HTTPS 443 to `management.azure.com` and `api.loganalytics.io`

Firewall rules, VPN tunnels, VPC peering, or private endpoints must be configured to allow this traffic.

**What happens if violated:** SIEM connector services will timeout on connection attempts. Health checks will report the SIEM as unreachable.

### 3.2 Agent-to-Internet for TI APIs

The agent requires outbound HTTPS (port 443) connectivity to:

- `www.virustotal.com`
- `api.abuseipdb.com`
- `api.crowdstrike.com` (or regional variant)
- The organization's MISP instance URL

If the agent runs in an air-gapped or restricted network, an HTTP proxy must be configured via `HTTPS_PROXY` environment variable.

**What happens if violated:** TI enrichment calls will fail with connection errors. The agent will log warnings and degrade gracefully, processing alerts without enrichment.

### 3.3 DNS Resolution Is Functional

The agent relies on standard DNS resolution for all external API endpoints. Custom DNS servers or split-horizon DNS must resolve these endpoints correctly.

**What happens if violated:** Connection failures with `Name or service not known` errors. Affects all external integrations.

## 4. Kubernetes Infrastructure

### 4.1 Cluster Is Provisioned and Operational

A Kubernetes cluster (EKS, AKS, GKE, or self-managed) must be provisioned, operational, and accessible via `kubectl` before deploying the agent. The cluster must meet these minimum requirements:

| Resource | Minimum | Recommended |
|----------|---------|-------------|
| Worker nodes | 3 | 5 |
| CPU per node | 4 vCPUs | 8 vCPUs |
| Memory per node | 8 GB | 16 GB |
| Kubernetes version | 1.28 | 1.30 |
| Storage class | Default (gp3 / managed-premium / pd-ssd) | SSD-backed |

**What happens if violated:** Helm install will fail or pods will remain in `Pending` state due to insufficient resources.

### 4.2 Ingress Controller Is Installed

An ingress controller (NGINX Ingress, AWS ALB Ingress, or Traefik) must be installed in the cluster to expose the agent's dashboard and API.

**What happens if violated:** The agent services will run but will not be accessible from outside the cluster.

### 4.3 Metrics Server Is Installed

The Kubernetes Metrics Server must be deployed for Horizontal Pod Autoscaler (HPA) to function.

**What happens if violated:** HPA will not scale pods based on CPU/memory utilization. Manual scaling will be required.

### 4.4 Persistent Volume Provisioner Is Available

A default StorageClass with a dynamic volume provisioner must be configured for PostgreSQL persistent volumes.

**What happens if violated:** PersistentVolumeClaim resources will remain in `Pending` state. Database pods will not start.

## 5. Database and Cache

### 5.1 PostgreSQL Is Accessible

A PostgreSQL 15 or 16 instance must be accessible from the agent pods. This can be a managed service (RDS, Cloud SQL, Azure Database) or a self-hosted instance. The agent requires:

- A dedicated database (e.g., `soc_analyst_agent`)
- A dedicated user with `CREATE`, `SELECT`, `INSERT`, `UPDATE`, `DELETE` permissions on that database
- Schema migrations are run by the agent on startup (requires `CREATE TABLE`, `ALTER TABLE`)

**What happens if violated:** The agent will fail to start. All API endpoints will return 503 errors.

### 5.2 Redis Is Accessible

A Redis 7.x instance must be accessible from the agent pods. Used for:

- Alert deduplication (bloom filters)
- Rate limit tracking for TI API calls
- Caching of enrichment results (TTL: 1 hour for IP/domain lookups)
- Celery task broker

**What happens if violated:** The agent will fail to start. Background task processing will be completely unavailable.

## 6. Security and Certificates

### 6.1 TLS Certificates Are Managed Externally

The agent does not provision or renew TLS certificates. Certificates for the agent's public endpoints must be managed via:

- cert-manager with Let's Encrypt (recommended for Kubernetes)
- AWS Certificate Manager (for ALB-terminated TLS)
- Azure Key Vault certificates
- Manually provisioned certificates loaded into Kubernetes Secrets

**What happens if violated:** HTTPS endpoints will use self-signed certificates or will not be available. Browser warnings will appear on the dashboard.

### 6.2 DNS Records Are Managed Externally

DNS records pointing to the agent's ingress endpoint (e.g., `soc-agent.internal.company.com`) must be created and managed by the organization's DNS team or automated via ExternalDNS.

**What happens if violated:** Users will need to access the agent by IP address or load balancer hostname, which is not compatible with TLS certificate validation.

## 7. Incident Response Integration

### 7.1 PagerDuty Service Is Configured

If PagerDuty integration is enabled, a PagerDuty service and integration key (Events API v2) must be provisioned before the agent is deployed.

**What happens if violated:** Critical alert escalation to PagerDuty will fail silently. The agent will log errors but continue operating.

### 7.2 Notification Channels Are Pre-Configured

Email (SMTP), Slack webhook, or Microsoft Teams webhook URLs must be provisioned if those notification channels are enabled in agent configuration.

**What happens if violated:** Notifications for that channel will fail. The agent will log delivery failures and continue processing.

---

## 8. Operational Assumptions

### 8.1 SOC Team Has Access to Agent Dashboard

SOC analysts must have network access (VPN, internal network) to the agent's web dashboard and be provisioned with user accounts.

### 8.2 Time Synchronization

All systems (agent, SIEM, databases) must be synchronized via NTP. Time drift greater than 5 seconds can cause alert correlation failures and JWT token validation errors.

### 8.3 Log Retention Policies Are Defined

The organization must define and configure log retention policies. The agent stores processed alert data in PostgreSQL indefinitely by default. Retention cleanup must be configured via the `ALERT_RETENTION_DAYS` environment variable or an external data lifecycle policy.
